Structuring Code (Because Future You Will Thank You) üõ†Ô∏è

Every ML project has three main phases:

  1. Data ‚Äì Get your data cleaned and ready.
  2. Train ‚Äì Build and test your model.
  3. Serve ‚Äì Deploy it for real-world use.

For deep learning projects, I stick to the following file structure:

  • main.py ‚Äì Orchestrates everything and provides a CLI interface.
  • data.py ‚Äì Handles preprocessing, PyTorch Dataset, and collate functions. If things get messy, I split it into smaller files.
  • model.py ‚Äì Defines the model, taking a config that includes hyperparameters (base model name, extra hidden layers, etc.).
  • trainer.py ‚Äì Handles data ingestion, loss calculations, and logs key metrics (precision, recall, false positives).

This structure has served me well over the years, but feel free to tweak it to your needs.

Side Quest: Clean Code (Your Teammates Will Love You) üßπ

Before we go any further, let‚Äôs have a moment of appreciation for clean code. It‚Äôs not just about looking nice‚Äîit helps teams actually collaborate and ship faster. Here‚Äôs my personal checklist:

  1. Keep PRs short ‚Äì Ideally <100 lines. Anything longer and reviewers start looking like this: Joma Tech video.
  2. Functions should be <15 lines ‚Äì One function, one job.
  3. Inject dependencies ‚Äì Pass objects into functions instead of creating them inside. This also makes writing tests so much easier!
  4. Files should be <150 lines ‚Äì If a file is getting too long, refactor and break it up (data.py ‚Üí data_process.py, data_warehouse.py).
  5. Name things well ‚Äì colour_image = image.convert("RGB") is way better than img = img.convert("RGB").

Of course, rules aren‚Äôt always absolute‚Äîjust do your best.

Parallelising Hypotheses üü∞

At my previous workplaces we used Argo WorkFlows (ArgoWF) to send our models into training. This is an abstraction on top of kubernetes that has worked well for us. However, there are other platforms that does similarly well in this space including AWS Steps, Ray Anyscale, KubeFlow etc.

What is important here regardless of the platform is that you spam the platform with all your hypotheses. In order to do so you need to structure your code to handle these cases and log metrics so that you can compare. These metrics need to be able to choose a winner and therefore should be independent of which model type is being tests.

In order to do this I use two tools:

  1. typer, a pretty cli tool for python that surpasses argparse.
  2. pydantic, for configurations and parsing in arguments from the cli.

Example:


class ModelConfig(pydantic.BaseModel):
        vision_model_name: str = "edgenext_small"
        embed_dims: int = 512

class TrainerConfig(pydantic.BaseModel):
        is_local: bool = False
        learning_rate: float = 1e-3
        batch_size: int = 8
        epochs: int = 2
        _model_config: ModelConfig = ModelConfig()

Local Execution: Sanity Check Before You Burn $$$ üí∏

Before we spam our experimentation platform with costly experiments that require massive GPUs, it is important to get a subset working locally.

When executing locally, we do not need to run the full training load. We simply need to make sure that each component of the pipeline from data transformations, to the model, to the training logic, work end to end. Therefore pushing a few batches (but not simply one) over 2 epochs should suffice. I usually choose 2 epochs over 1 to make sure that I don‚Äôt see any funny errors popping up at the end of training.

I have logic in my training code similar to the following:

if trainer_config.is_local:
        df = df.iloc[:200]
        trainer_config.epochs = 2

Tracking Experiments: Logs or It Didn‚Äôt Happen üëÄ

Logging is important so that we can compare each run. Whilst I generally use wandb, there are plenty of other commercial and open source alternatives available. As far as logging goes there are two things that are important:

  1. Log a metric that makes product sense.
  2. If your problem is based on images, text, or other media, log visual examples.

In terms of logging a metric, this cannot be simply the loss. Loss does not say anything about how ‚Äúaccurate‚Äù your model is. While precision and recall are good metrics to track, keep in mind that we can adjust precision at the expense of recall (and vice versa). When it comes to classification (multi-label or otherwise) average AUROC is a good metric. Keep in mind this was only an example of being creative with your metrics, and by all means adjust this to what makes sense.

At the end of training it is important to log the results of your model so that you can visually inspect the results. With a recent project that we did, we logged the false positives and false negatives. The surprising result that we observed was that the false positives were more often than not mis labelled as opposed to the model getting it wrong.

Summary üöÄ

  • Run many experiments, not just one at a time.
  • Move from notebooks to Python files once initial exploration is done.
  • Structure your ML projects into data, train, and serve phases.
  • Keep code clean and modular to boost collaboration.
  • Use Argo Workflows (or similar) to parallelise training jobs.
  • Always sanity check locally before running expensive cloud experiments.
  • Log everything so you can compare and debug later.
